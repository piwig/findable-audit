## 2. Scoring model

Backward-compatible weighted model. Keep `score: number` (0–100); add `grade` and `familyScores`.

**Per check** (unchanged `makeResult`): `pass` → maxPoints, `warn` → `floor(maxPoints/2)`, `fail` → 0, `skip` → excluded.

**Per family**: `sub_i = earned_i / max_i` over that family's **non-skip** checks. If a family has zero non-skip checks (`max_i === 0`) it is **excluded** and its weight is redistributed proportionally over the remaining families.

**Overall**:

```
included = families with max_i > 0
W        = Σ weight_i         (i ∈ included)
score    = round( 100 * Σ (weight_i * sub_i) / W )   (i ∈ included)
```

**Letter grade** (`gradeOf(score)`): `A` ≥ 90 · `B` ≥ 80 · `C` ≥ 70 · `D` ≥ 60 · `F` < 60.

**Invariant (migration-critical)**: when every non-skip check passes, every `sub_i = 1.0`, so `score = round(100 * W / W) = 100` for *any* weights. The existing "perfect site → 100" test therefore still holds. ⚠️ **But** new always-on static checks (headings-outline, charset, favicon, landmarks, html-lang, img-dimensions, …) will *run* on the perfect-site fixture and must *pass* there, or the fixture must be extended. Tests asserting exact *intermediate* scores will change with weighting and must be updated; only the perfect-100 invariant is contractual.

**`AuditReport` additions**:

```ts
export interface AuditReport {
  url: string;
  score: number;                 // kept, now weighted
  grade: 'A'|'B'|'C'|'D'|'F';    // new
  familyScores: { family: Family; score: number; weight: number; earned: number; max: number }[]; // new
  sampledPages: string[];
  results: CheckResult[];
}
```

`FAMILY_WEIGHTS: Record<Family, number>` lives next to the runner; the weighted math replaces the flat `earned/max` in `runner.ts`.

## 4. Gap analysis (skeptical top-agency lead) — gaps folded in

After drafting, I audited the catalogue as an agency lead hunting for a missing *automatable* check. Folded in:

- **HTTP `Link: rel="canonical"` header** — canonical can be delivered by header, not just `<link>`; `canonical` now honors both.
- **Duplicate / multiple canonical tags** — `canonical` fails on >1 canonical, not just on absence.
- **HSTS quality** (`includeSubDomains`/`preload`) — folded into `hsts` as bonus grading.
- **Preview-starving directives** (`nosnippet`/`max-snippet:0`) as a *positive* GEO signal — dedicated `snippet-preview-directives`.
- **`theme-color`** and **apple-touch-icon** — folded into `favicon`.
- **WebSite SearchAction** (sitelinks searchbox) — `sd-website-searchaction`.
- **Wikipedia/Wikidata grounding** as a distinct, higher-value signal than generic sameAs — `sd-entity-grounding`.
- **Soft-404 via a random-path probe** and **custom-404 recoverability** — `soft-404` + `custom-404`.
- **Meta-refresh redirects** as a hidden redirect class — `meta-refresh`.
- **Near-duplicate bodies across the sample** (shingling) — `content-uniqueness`.
- **Merchant depth** (`gtin`/`mpn`, `priceValidUntil`, `shippingDetails`) — noted as bonus grading inside `sd-product`.

Explicitly **deferred / dropped as not reliably automatable** from a static fetch (see §7): color contrast, tap-target size, and any layout/CSS-computed a11y (need rendering); external outbound-link liveness at scale (rate-limit noise — we check *presence*, not HTTP status, in `outbound-citations`); microdata/RDFa-vs-JSON-LD conflict (low value, rare); HTML validity / duplicate-ID linting (lenient parser, low signal); JS-rendered content correctness (the tool is static by design — surfaced instead as `content-without-js`).

## 5. Core Web Vitals plan (performance family, Phase D)

New module `packages/cli/src/perf/psi.ts`; new family `performance`.

### Flags (mirror the existing `--indexnow-key` pattern in `src/index.ts`)

- `--cwv` (boolean, default off) — opt into the slow (~15–30s) PageSpeed Insights call. Without it, all `cwv-*`, `lighthouse-perf`, `lab-*` checks return `skip`. Static heuristics always run.
- `--psi-key <key>` (string) — Google PSI/CrUX API key. **Required for reliable results**: the v5 *keyless* endpoint returns **HTTP 429** (rate-limited) in practice.
- `--psi-strategy <mobile|desktop>` (string, default `mobile`).

### Key handling / behavior matrix

| invocation | behavior |
|---|---|
| no `--cwv` | all PSI/CRUX checks `skip` ("run with `--cwv --psi-key <key>` to measure Core Web Vitals"). |
| `--cwv`, no `--psi-key` | one keyless attempt; on 429 / no data → all PSI/CRUX checks `skip` ("PageSpeed Insights rate-limited without a key; pass `--psi-key`"). |
| `--cwv --psi-key` | one PSI call, response cached and shared by every PSI/CRUX check. Field metrics present → grade p75. Field absent (low-traffic URL) → field checks `skip` ("no CrUX field data"), lab checks (`lighthouse-perf`,`lab-*`, lab-fallback in `cwv-lcp`) grade. |

Single call: `GET https://www.googleapis.com/pagespeedonline/v5/runPagespeed?url=<final>&strategy=<mobile|desktop>&category=performance&key=<key>`, native `fetch`, honoring the run's `AbortSignal`/timeout. Zero new deps. Parse:
- Field: `loadingExperience.metrics.{LARGEST_CONTENTFUL_PAINT_MS, CUMULATIVE_LAYOUT_SHIFT_SCORE, INTERACTION_TO_NEXT_PAINT, EXPERIMENTAL_TIME_TO_FIRST_BYTE}.percentile` (p75) + `.category`, and `loadingExperience.overall_category`; url-level preferred, `originLoadingExperience` fallback (flag which was used in the message).
- Lab: `lighthouseResult.categories.performance.score`; `lighthouseResult.audits['largest-contentful-paint'|'first-contentful-paint'|'total-blocking-time'|'server-response-time'|'speed-index'].numericValue`.
