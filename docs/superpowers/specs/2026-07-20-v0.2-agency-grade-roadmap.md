# findable-audit v0.2 — "agency-grade" roadmap

Date : 2026-07-20 (session autonome nocturne)
Statut : roadmap maître — exécution autonome autorisée par l'utilisateur
("mets en œuvre les bonnes idées sans me poser de questions").

## Vision

Rendre `findable-audit` au moins aussi complet qu'un audit d'agence SEO/GEO
spécialisée : couverture maximale de l'optimisation d'un site (SEO technique,
on-page, données structurées, GEO/IA, **Core Web Vitals**, accessibilité,
sécurité/confiance), un scoring lisible et actionnable, et un **site web** pour
lancer des audits facilement (hébergé sur le VPS Hetzner).

Contraintes conservées : Node ≥ 20, ESM, cross-platform strict, `process.exitCode`,
zéro dépendance runtime lourde (le CLI reste léger ; les Core Web Vitals passent
par l'API PageSpeed Insights via `fetch`, pas de Chromium embarqué).

## Phases

- **A. Recherche & design** (workflow multi-agents) → spec exhaustif des nouveaux
  checks (SEO technique, on-page, données structurées riches, GEO/E-E-A-T,
  performance/CWV, accessibilité, sécurité), avec pour chacun : famille,
  vérifiabilité (HTML statique / en-tête HTTP / API PSI / multi-pages), seuils,
  points, message, fix. Livrable : `2026-07-20-v0.2-checks-spec.md`.
- **B. Modèle de scoring** : passage à un scoring par **catégories pondérées** +
  note lettre (A–F) + sous-scores par catégorie, pour absorber ~40+ checks sans
  diluer le signal. Rétro-compat : champ `score /100` conservé.
- **C. Nouveaux checks statiques/HTTP** (par lots, TDD + revue) : head/meta
  complets, Open Graph/Twitter riche, JSON-LD par type (Article/Product/FAQ/
  Breadcrumb/Organization/WebSite+SearchAction), hiérarchie de titres, fraîcheur
  de contenu, densité/thin content, liens internes (profondeur, ancres), images
  (dimensions pour CLS, lazy, formats), canonical/trailing-slash/www, 404 custom,
  chaînes de redirection, sitemap riche (lastmod), en-têtes de sécurité (HSTS,
  X-Content-Type-Options, CSP), langue/hreflang/x-default.
- **D. Core Web Vitals** (`src/perf/`, flag `--cwv`) : appel API PageSpeed
  Insights v5 (keyless + `--psi-key` optionnel) → score Lighthouse, LCP, CLS,
  INP, FCP, TTFB, TBT (lab) + données terrain CrUX si dispo. Nouvelle famille
  "performance" ; heuristiques statiques (poids page, scripts bloquants,
  images sans dimensions) actives par défaut, métriques terrain sur `--cwv`.
- **E. Site web de test** (`apps/web`) : petit serveur Node (zéro dépendance)
  servant un frontend (saisie d'URL → lance l'audit → rend le rapport HTML +
  lien partageable), file d'attente simple, cache. Testé localement.
- **F. Déploiement Hetzner** : sous-domaine dédié + service systemd isolé +
  bloc nginx + Cloudflare/Let's Encrypt, en suivant le patron des autres sites,
  SANS toucher aux services existants. Si risque de collision détecté → préparer
  et laisser `PENDING` pour validation.
- **G. Docs, tests, revue finale, ship** : README/guides EN+FR, CI verte,
  revue adversariale de toute la branche.

## Méthode d'exécution (utilisateur endormi)

- Chaque phase : design → implémentation TDD → **revue adversariale par subagents**
  (remplace l'approbation utilisateur) → merge dans la branche.
- Ledger de progression maintenu dans `.superpowers/sdd/progress.md`.
- Actions sortantes (push, déploiement VPS) : le push régulier de la branche est
  autorisé (repo perso OSS) ; le déploiement VPS suit le principe de prudence ci-dessus.
- Rien n'est mergé dans `main` sans une revue finale verte ; un récapitulatif
  complet sera prêt au réveil.
